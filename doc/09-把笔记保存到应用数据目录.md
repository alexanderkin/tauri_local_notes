# 第 9 步：把笔记保存到应用数据目录

## 1. 这一步的目标

把笔记文件从项目源码目录移走，改为保存到应用自己的数据目录。

这样做的原因是：

- 保存笔记时不再触发开发模式重启
- 更符合真实桌面应用的数据存放方式

## 2. 之前为什么会重启

之前笔记保存到了：

```text
src-tauri/notes.txt
```

而你运行的是开发模式：

```powershell
npm.cmd run tauri dev
```

开发模式会监视项目目录里的文件变化。

所以每次程序写入 `src-tauri/notes.txt` 时，开发工具都会误以为你改了项目文件，于是自动重新编译、重启应用。

## 3. 正确做法是什么

用户数据不应该长期放在项目源码目录里。

更合理的做法是：

- 项目代码放在项目目录
- 运行时数据放在应用数据目录

这样可以把“开发文件”和“用户数据”分开。

## 4. 这一步你做了什么

你在 Rust 里新增了一个统一获取笔记文件路径的函数，大致思路是：

1. 向 Tauri 获取应用数据目录
2. 如果目录不存在，就先创建
3. 在这个目录下拼出 `notes.txt`

保存和读取笔记都改成使用这个新路径。

## 5. 这一步里遇到的报错

你遇到的报错是：

```text
no method named `path` found for reference `&AppHandle`
```

这不是逻辑错误，而是少导入了一行：

```rust
use tauri::Manager;
```

## 6. 为什么加 `use tauri::Manager;` 就好了

因为 `app.path()` 这个能力属于 Tauri 的 `Manager`。

Rust 的很多能力不是“自动全部可用”的，而是需要你先导入对应内容。

所以：

- 方法本身是存在的
- 只是你要先把对应能力引进来

## 7. 这一步完成后的结果

修复完成后，你已经确认了两件事：

- 报错没了
- 保存笔记时不再触发应用重启

这说明数据存放位置已经更合理了。

## 8. 这一步完成后你学到了什么

- 开发模式会监视项目目录中的文件变化
- 用户数据不适合长期保存到源码目录
- Tauri 可以提供应用自己的数据目录
- Rust 报错里的 `help:` 往往很重要，很多时候会直接提示解决方案

## 9. 这一步的小复盘

你刚刚做的事情不是新功能，而是一次很重要的工程修正。

它在整个项目里的作用是：
让你的 App 更接近真实桌面应用的做法，而不是只停留在教学演示层面。

这一步最需要记住的是：

- 运行数据和源码目录最好分开
- `app.path()` 这类能力有时需要额外导入
- 遇到 Rust 报错时，先看错误下面的 `help:` 提示

## 10. 下一步要做什么

下一步我们会继续之前暂停的功能：

- 读取已经保存的笔记
- 在页面上显示出来

这样你的笔记本就会形成完整闭环：

- 能输入
- 能保存
- 能看到已保存内容
