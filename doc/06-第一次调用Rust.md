# 第 6 步：第一次从前端调用 Rust

## 1. 这一步的目标

让前端页面真正调用到 Rust 代码，并拿到 Rust 返回的结果。

这是 Tauri 非常关键的一步，因为它体现了桌面应用和普通网页的一个核心区别：
页面不只是自己显示内容，还可以请求本地程序能力。

## 2. 你刚刚做了什么

你修改了两个文件：

```text
tauri-notes/src/main.js
tauri-notes/src-tauri/src/lib.rs
```

你完成的事情是：

- 前端点击按钮
- 把标题传给 Rust
- Rust 返回一段文字
- 页面显示这段文字

## 3. `main.js` 在这里做了什么

前端主要做了 3 件事：

1. 读取输入框内容
2. 点击按钮时调用 Rust
3. 把 Rust 返回结果显示到页面上

这里最重要的一行是：

```javascript
const result = await invoke("greet", { title });
```

你可以先这样理解：

- `"greet"` 是要调用的功能名
- `{ title }` 是传给 Rust 的数据
- `result` 是 Rust 返回给前端的结果

## 4. `lib.rs` 在这里做了什么

Rust 这边最重要的内容有两部分。

### 第一部分：定义一个可以被前端调用的函数

```rust
#[tauri::command]
fn greet(title: &str) -> String {
    format!("Rust 已收到标题：{}", title)
}
```

这里可以先这样理解：

- `#[tauri::command]`：告诉 Tauri，这个函数可以给前端调用
- `title: &str`：接收前端传来的标题文字
- `String`：把结果文字返回给前端

### 第二部分：把这个函数注册到 Tauri

```rust
.invoke_handler(tauri::generate_handler![greet])
```

这句的意思是：
把 `greet` 这个功能挂到 Tauri 上。

如果不注册，前端即使写了调用代码，也找不到这个功能。

## 5. 为什么这一步很重要

因为从这一步开始，你已经不是只会做一个前端页面了。

你已经走通了 Tauri 最关键的链路：

- 前端界面
- 前端发起调用
- Rust 接收数据
- Rust 返回结果
- 页面显示结果

后面做保存文件，本质上就是把现在这个简单示例升级成真正的本地功能。

## 6. 这一步完成后你学到了什么

- 知道前端可以通过 `invoke` 调用 Rust
- 知道 Rust 函数需要用 `#[tauri::command]` 标记
- 知道 Rust 功能需要注册，前端才能调用
- 知道 Tauri 的前后端通信是怎么连起来的

## 7. 常见错误和排查方法

### 情况 1：提示找不到命令

排查方向：

- 检查 Rust 函数上有没有 `#[tauri::command]`
- 检查有没有在 `invoke_handler` 里注册
- 检查前端调用名字和 Rust 函数名字是否一致

### 情况 2：点击按钮没有结果

排查方向：

- 检查按钮点击事件是否还在
- 检查 `invoke(...)` 是否写在点击事件里
- 检查页面消息区域是否正常更新

### 情况 3：Rust 编译报错

排查方向：

- 检查 `lib.rs` 代码有没有少括号或少逗号
- 检查函数返回类型和实际返回值是否一致

## 8. 这一步的小复盘

你刚刚做的事情是：
让前端第一次成功调用了 Rust。

它在整个项目里的作用是：
打通了 Tauri 最关键的通信链路。

这一步最需要记住的是：

- 前端用 `invoke` 调 Rust
- Rust 用 `#[tauri::command]` 暴露功能
- Rust 还要注册这个功能，前端才能找到

- Rust 把笔记写到本地文件
