# 第 23 步：给笔记增加唯一 ID

## 1. 这一步的目标

给每条笔记增加一个唯一 ID，让删除和后续编辑功能更稳定。

## 2. 为什么需要 ID

之前删除笔记时，主要依赖：

- 位置编号
- 或者时间、标题、内容组合

这些方式都不够稳定。

比如：

- 搜索后显示顺序可能变化
- 两条笔记可能内容一样
- 位置编号会随着删除、排序发生变化

所以更合理的做法是：
每条笔记保存时都有自己的唯一 ID。

可以把 ID 理解成：
每条笔记自己的身份证号。

## 3. Rust 数据结构怎么变了

之前一条笔记是：

```rust
struct Note {
    time: String,
    title: String,
    content: String,
}
```

后来升级成：

```rust
struct Note {
    #[serde(default)]
    id: String,
    time: String,
    title: String,
    content: String,
}
```

其中：

- `id`：每条笔记的唯一编号
- `#[serde(default)]`：兼容旧数据，如果旧笔记没有 ID，就先用默认值

## 4. 为什么会遇到旧数据问题

升级结构后，新代码要求每条笔记都有 `id`。

但旧的 `notes.json` 里，之前保存的笔记没有这个字段。

所以一开始出现了：

```text
missing field `id`
```

这是数据结构升级时很常见的问题。

## 5. 怎么兼容旧数据

先用：

```rust
#[serde(default)]
```

让旧数据即使缺少 `id` 也能被读取。

然后在读取笔记时检查：

- 如果某条笔记的 `id` 是空的
- 就自动生成一个新 ID
- 再把更新后的数据写回 `notes.json`

这就是一个简化版的数据迁移。

## 6. 删除方式怎么升级了

之前删除是按位置：

```rust
delete_note(index)
```

现在删除是按 ID：

```rust
delete_note(id)
```

前端按钮里保存的是：

```html
data-id="..."
```

点击删除时，前端把 ID 传给 Rust。

Rust 再根据 ID 找到对应笔记并删除。

## 7. 这一步完成后你学到了什么

- ID 比位置编号更稳定
- 数据结构升级要考虑旧数据
- `#[serde(default)]` 可以处理缺失字段
- 读取数据时可以顺手做一次旧数据迁移
- 删除、编辑这类操作最好基于稳定 ID

## 8. 这一步的小复盘

你刚刚完成的是一次非常真实的项目升级：

- 修改数据结构
- 兼容旧数据
- 修改删除逻辑
- 修改前端传参方式

它在整个项目里的作用是：
让数据管理能力更稳定，为后续编辑笔记打基础。

这一步最需要记住的是：

- `index` 会变，`id` 应该稳定
- 新版本代码要考虑旧版本数据
- 数据迁移是项目变复杂后常见的工作
