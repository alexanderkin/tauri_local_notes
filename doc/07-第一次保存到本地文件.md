# 第 7 步：第一次保存到本地文件

## 1. 这一步的目标

让页面里的笔记内容真正写入本地文件。

这一步完成后，我们的项目第一次拥有了“真实数据落地”的能力，不再只是页面演示。

## 2. 你刚刚做了什么

你修改了这两个文件：

```text
tauri-notes/src/main.js
tauri-notes/src-tauri/src/lib.rs
```

你完成的事情是：

- 前端读取标题和内容
- 前端把数据传给 Rust
- Rust 把内容写入本地文件
- 页面显示保存成功

最后你也确认了：

- 页面显示保存成功
- 项目目录中出现了 `notes.txt`

## 3. 这一步里最重要的 Rust 代码

### 使用文件功能

```rust
use std::fs;
```

意思是：
使用 Rust 自带的文件读写功能。

### 写入文件

```rust
fs::write("notes.txt", text)
```

意思是：
把 `text` 这段文字写进 `notes.txt` 文件。

如果这个文件原本不存在，就会创建它。

## 4. 为什么页面能显示“保存成功”

因为 Rust 函数成功执行后，会返回一段文字给前端。

前端再把这段文字显示到页面上。

也就是说，这一步同时练到了两件事：

- 前后端通信
- 本地文件写入

## 5. 当前版本的一个特点

现在这份代码使用的是：

```rust
fs::write(...)
```

它的特点是：
**每次写入都会覆盖原来的文件内容。**

也就是说，如果你保存第二次，前一次的内容就会被替换掉。

这不是错误，而是我们当前阶段故意采用的最简单版本。

原因是：

- 先学会“能写进去”
- 再学会“连续保存多条”

## 6. 这一步完成后你学到了什么

- 知道 Rust 可以直接操作本地文件
- 知道前端可以把表单内容传给 Rust
- 知道 Tauri 项目已经能做真实本地功能，而不只是显示界面

## 7. 常见错误和排查方法

### 情况 1：页面显示失败

排查方向：

- 检查 `invoke("save_note", ...)` 名字是否和 Rust 一致
- 检查标题和内容是否都传过去了
- 检查 Rust 函数有没有注册

### 情况 2：Rust 编译报错

排查方向：

- 检查 `use std::fs;` 是否存在
- 检查函数返回类型是否和返回值一致
- 检查括号和分号是否缺失

### 情况 3：没看到保存文件

排查方向：

- 看当前项目实际运行目录
- 在项目目录和 `src-tauri` 目录里都检查一下
- 确认页面提示是否真的是保存成功

## 8. 这一步的小复盘

你刚刚做的事情是：
让笔记真正保存到了本地文件。

它在整个项目里的作用是：
把项目从“交互演示”推进成了“真正能处理本地数据的桌面应用”。

这一步最需要记住的是：

- Rust 可以负责本地文件操作
- 前端负责采集输入和显示结果
- 现在已经能保存，但当前版本是覆盖保存

## 9. 下一步要做什么

下一步我们会优化保存方式：

- 不再覆盖旧内容
- 每保存一次，就在文件后面追加一条新笔记

这样它才更像一个真正的笔记本。
